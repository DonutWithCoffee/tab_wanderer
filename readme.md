# tab_wanderer

Chrome extension для автоматического отслеживания заказов в админке amperkot.ru.

---

## Статус проекта

Текущий статус проекта: **stable core / engine**.

Это уже не прототип. Текущее ядро считается стабильной базой, поверх которой развиваются rules layer и feature layer.

Текущая версия: **v0.9.4**

---

## Что уже входит в stable core

Core включает:

- стабильный worker lifecycle
- изоляцию worker-вкладки
- recovery после restart / reload
- watchdog
- baseline без ручного вмешательства
- deterministic order processing
- rules engine перед уведомлением
- открытие заказа по клику на уведомление

---

## Архитектурная модель

Проект делится на 3 слоя.

### Core

Меняется только при реальной необходимости.

Сюда входят:

- worker lifecycle
- tab isolation
- recovery logic
- watchdog
- baseline / storage
- `processOrders()` как orchestration layer
- точка входа rules engine

### Rules Layer

Основной расширяемый слой:

- `notification-rules.js`
- фильтрация уведомлений
- логика принятия решения `notify / ignore`
- нормализация значений
- составные правила

### Feature Layer

Надстройка над ядром:

- конфигурация
- настройки пользователя
- типы уведомлений
- вспомогательный UI

---

## Как работает

1. создаётся изолированная worker-вкладка
2. content script собирает заказы
3. baseline и изменения определяются детерминированно через hash
4. перед уведомлением применяется rules engine
5. уведомление либо отправляется, либо прозрачно игнорируется по правилу
6. при клике на уведомление открывается конкретный заказ

---

## Текущий приоритет

Текущий следующий шаг после стабилизации core:

1. собрать production-набор notification rules
2. затем перейти к user configuration
3. затем довести проект до Definition of Done для v1.0

Принцип развития проекта:

**stable core → controlled rules expansion → user configuration → 1.0**

## Roadmap

Каждый следующий актуальный message 51 должен содержать этот roadmap.

### Этап 1 — Notification Hardening

- правила по статусам
- правила по доставке
- расширение правил по оплате
- составные правила
- нормализация значений
- прозрачное логирование правил

### Этап 2 — Event Model

Различать:

- новый заказ
- изменение заказа
- смена статуса
- уведомления по типу события

### Этап 3 — Configuration

- централизованный конфиг
- включение / выключение правил
- настройка уведомлений пользователем
- расширяемость без изменения core

### Этап 4 — Observability

- история событий
- история срабатывания правил
- диагностика без debug UI

## Правила разработки

Перед любыми изменениями всегда:

1. сначала анализ текущей реализации
2. потом решение
3. потом код

Без анализа код не писать.

При ревью кода каждое изменение проверяется по 3 вопросам:

1. соответствует ли оно roadmap
2. не ломает ли core
3. нельзя ли вынести это в rules layer или feature layer

Любое изменение должно быть привязано к этапу roadmap.

### Правила внесения изменений

- работать через VS Code terminal
- не плодить лишние коммиты
- перед коммитом обязательно делать ревью
- при изменениях показывать полный код редактируемого файла
- перед новым кодом показывать старый редактируемый блок
- избегать временных решений
- не добавлять debug UI

## Архитектурные ограничения

Запрещено:

- искать worker через поиск обычных admin-вкладок по URL
- переиспользовать чужие вкладки
- завязывать core на UI страницы
- строить логику на ручном вмешательстве пользователя
- усложнять архитектуру без причины

Обязательно:

- worker определяется через marker + tabId
- поведение должно быть детерминированным
- логика должна быть прозрачной
- recovery должен быть безопасен после restart браузера / ПК

## Definition of Done для v1.0

Версия 1.0 считается готовой только если выполнены все условия ниже.

### A. Core stability

- worker создаётся и работает без ручного вмешательства
- worker не дублируется
- worker корректно восстанавливается после restart браузера
- worker корректно восстанавливается после restart ПК
- watchdog не даёт ложных пересозданий
- нет race condition в startup / recovery сценариях

### B. Deterministic processing

- baseline полностью автоматический
- изменение заказа определяется предсказуемо
- нет ложных уведомлений из-за смешения core и business logic
- состояние не теряется после restart

### C. Notification system

- все уведомления проходят через единый rules engine
- rules engine поддерживает фильтрацию по:
  - статусу
  - доставке
  - оплате
- правила расширяются без переписывания core
- при клике на уведомление открывается новая вкладка с конкретным заказом

### D. User configuration

- пользователь может настраивать уведомления
- пользователь может включать / выключать правила или группы правил
- настройки не ломают core
- UI настроек минимален и понятен

### E. UX minimum

- popup содержит только необходимое управление
- START / STOP / status работают предсказуемо
- нет debug-кнопок и тестового мусора в UI

### F. Operational quality

- нет лишних save / reload без причины
- логи позволяют разбирать реальные проблемы
- поведение расширения не требует постоянного контроля пользователем

### G. Release quality

- `manifest.json`, `version.js`, `popup` и `README` синхронизированы
- git history чистая и production-level
- релиз можно отдать коллегам без оговорки «это пока нестабильно»

---

## Что не входит в обязательный v1.0

Необязательно для 1.0, если не потребуется отдельно:

- сложный UI-конструктор правил
- история событий с отдельным экраном
- продвинутая аналитика
- сложные allow / deny priority systems
- всё, что усложняет core без прямой пользы

---

## Что реализовано в v0.9.4

В рамках этапа **Notification Hardening** в текущей версии уже есть:

- правила игнора по статусам
- правила игнора по доставке
- расширенные правила игнора по оплате
- составные ignore rules
- нормализация строковых значений
- прозрачная причина rule match в результате rules engine

Это изменение не требует переписывания core и остаётся внутри rules layer.
